## Honest Feedback (Session-Specific Friction)

**Friction 1: Schema Mismatch Pattern**  
The landing page was written without checking the actual copy.json structure. This is a common pattern in new projects—template developers make assumptions about data shapes. In this case, it delayed deployment. A simple schema validation (TypeScript strict mode or Zod) would have caught this at dev time, not deploy time.

**Friction 2: Two-Deploy Cycle**  
The first fix (content import) seemed to solve the problem but didn't. Would have saved time with a more systematic "read all files first" approach before declaring victory. The second failure forced the deeper inspection that revealed the real issues.

**Friction 3: Worktree Context Shift**  
Switching from shared checkout to worktree required tool learning (EnterWorktree behavior, path changes). This was fine but represents operational overhead. The safety it provides is worth it, but worth noting for future optimization.

## Lessons Learned

**Lesson 1: Schema-First Debugging**  
When diagnosing data-driven failures, always read the data schema (the source of truth) before reading the consumer code. Patterns like empty objects can mislead. Source-first reasoning is faster.

**Lesson 2: Build Errors Point to Line Numbers**  
Vercel's build errors included exact file/line references ("at a.s.title"). This specificity is gold for debugging. Always scan error messages for coordinates before wild guessing.

**Lesson 3: Multiple Issues Can Hide Behind One Error**  
The first error (empty objects) masked three more issues (field names, property names, data structure). Fixing one doesn't guarantee success. After a first fix, run the full pipeline again before celebrating.

**Lesson 4: Worktree Isolation Enables Confidence**  
Fixing production code in an isolated worktree, committing, and testing before merge reduces anxiety. The ceremony is worth the safety.
